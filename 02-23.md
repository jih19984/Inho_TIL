# 2026-02-23 TIL

## 오늘 배운 것 (Today I Learned)

### 1) 시스템 확장성/성능 기본 패턴 정리
- **Scale-up(수직 확장)**: 서버 한 대를 더 좋은 CPU/RAM/디스크로 업그레이드
  - 구조는 단순하지만 한계가 명확하고, 1대 장애 시 전체 영향이 큼(SPOF)
- **Scale-out(수평 확장)**: 노드를 여러 대 늘리고 로드밸런싱
  - 트래픽 분산 및 가용성 측면에서 유리, 운영은 상대적으로 복잡해짐(상태/세션 동기화)
- 결론: 초기에는 **규모 대비 가성비 높은 방식인 Scale-out 중심** 설계를 우선 검토

### 2) DB 최적화 (읽기/쓰기 비중 관점)
- 대부분 서비스는 `Read`가 `Write`보다 많음
- **Master-Replica 구조**
  - Master: 쓰기 전용
  - Replica: 읽기 전용 + 읽기 부하 분산
- 고가용성(HA)을 위해 Replica 다중화 및 장애 시 Failover 동작 고려

### 3) 캐시 전략
- DB에 직접 가기 전에 메모리 캐시에서 먼저 조회해 응답 속도 개선
- 적용 대상: 자주 조회되지만 변경이 적은 데이터
- 운영 포인트
  - TTL(유효시간)
  - 데이터 변경 시 캐시 무효화 일관성
  - Eviction 정책(LRU/LFU/FIFO)

### 4) CDN(정적 자원 캐시)
- 정적 파일(CSS/JS/이미지) 배포를 사용자 근처에서 빠르게 전달
- 장점: TTFB 개선 + 원천 서버 부하 감소
- 운영 주의: 버전 기반 무효화/캐시 갱신 정책, 트래픽이 낮으면 과투자 주의

### 5) 비정형 데이터 + 직렬화
- 비정형 데이터: 로그, 이미지, 이벤트 스트림처럼 스키마 고정이 약한 데이터
- 직렬화: 메모리 객체를 바이트 스트림으로 바꿔 전송/저장
- 형식 선택: 빠른 처리 목적이면 가볍고 간단한 포맷, 장기 저장/호환성은 스키마 관리형 포맷 고려
