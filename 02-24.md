# 2026-02-24

## 1) 추정과 스프린트 핵심
- 애자일은 **빠르게 끝내는 것**이 아니라 **빠르게 피드백 받아 개선**하는 방법이다.
- 추정은 일정 보증이 아니라 **팀 속도를 파악하기 위한 장치**다.
- 불확실성이 크면 추정이 빗나가므로, 이터레이션을 반복할수록 정확도가 올라가야 한다.
- 스프린트가 계획보다 늦어졌더라도 실패가 아니라, **회고에서 개선 포인트를 찾는 신호**로 본다.

## 2) 스토리 포인트
- 팀의 개발 속도/역량을 가늠하는 수치
- 당초 예상치가 곧 약속이 아님
- 계획: 백로그 정리 → 스프린트 계획 → 개발 → 산출물 리뷰(Sprint Review) → 회고(Sprint Retrospective)
- 스케일: 피보나치/수정 피보나치/2의 제곱/1~5 등 팀 규칙에 맞게 사용
- Planning Poker: 팀원마다 다르게 판단할 수 있으므로 의견 차이를 공유해 합의 기준을 맞춘다.

## 3) 번다운이 기린/박쥐 형태가 되는 이유
- 현실에서는 번다운이 매끈히 내려가지 않는 경우가 일반적이다.
- 기린형/박쥐형은 “잘못”이 아니라 **병목과 의존성의 가시화**다.
- 대응: 
  - 시작 가능한가를 따지는 Definition of Ready 보강
  - 큰 항목을 더 작은 배달 가능한 단위로 쪼개기
  - 블로커를 숫자/책임자 기준으로 관리

## 4) 데일리 스크럼 운영 방식
- 회의는 짧게(매일):
  1. 어제 한 일
  2. 오늘 할 일
  3. blocker
- blocker가 나오면 그 자리에서 함께 해결할 Action Item(담당자·기한)을 정한다.
- 데일리 스크럼은 피로해지면 무의미해지므로, 필요한 만큼만 유연하게 운영한다.

## 5) 리뷰와 회고의 역할
- 스크럼 팀 성과는 코드 리뷰/회고로 높아진다.
- 목표는 비난이 아니라 **실수 재발 방지와 프로세스 개선**.

## 6) 사일로와 Bus Factor
- 사일로(부서/개인 단위 고립)는 팀 속도를 떨어뜨린다.
- Cross-functional 협업으로 서로의 영역을 넘나들고, 지식 공유를 습관화해야 한다.
- Bus Factor를 높이려면: 페어 프로그래밍, 교차 리뷰, 공유 문서/결정 로그가 필수다.
- 특정 인력이 없어져도 팀이 멈추지 않게 운영 구조를 만든다.

## 7) 칸반 관점의 WIP 제어
- Kanban은 To do / WIP / Done 보드와 WIP 제한이 핵심이다.
- 새 작업은 기존 WIP를 끝내야 시작 가능하도록 한다.
- WIP 한도를 팀 상황에 맞춰 설정해 과부하를 막고 협업을 유도한다.
- 실무 기준으로는 팀 사이즈와 스프린트 복잡도에 맞춘 초기 WIP를 정한 뒤 조정한다.

## 9) 프론트엔드-백엔드 연동 정리
- React Native/웹 구조는 `요청(클라이언트) -> 가공(백엔드) -> 응답(화면)`로 흘러간다.
- 정적 문서 페이지(Wikipedia 형태)에서 동적 서비스로 바뀌며, UX/인터랙션/데이터 가공 로직이 프론트/백엔드 분리를 요구한다.
- AJAX는 페이지 전체를 새로고침하지 않고 JSON 기반으로 일부만 갱신하는 방식이다.
- SPA는 AngularJS(2009/2010), React(2013), Angular 2(2016), Vue(2014~) 이후 본격 확산되었고, 현재는 페이지 단위가 아니라 상태 기반 화면 갱신이 기본이 됐다.
- JavaScript는 웹 페이지 동적 제어 핵심 언어이고, 함수는 값처럼 다뤄지는 일급 객체다.
- 원시값(primitive)은 단일 값(`string`, `number`, `boolean`, `undefined`, `null`, `symbol`, `bigint`)이고, 화살표 함수는 짧은 문법과 lexical this가 특징이다.

## 10) 슬롯 배치 API/UX 기획 정리
- 슬롯 API는 저장 버튼 기반 일괄 반영 방식으로 정리: `repo_id + slot_index + planet_type` 동기화.
- 슬롯 인덱스는 은하계 슬롯(0~7) 위치를 의미, 행성 타입은 8개 고정(수성/금성/지구/화성/목성/토성/천왕성/해왕성).
- 좌표/위치 데이터(`x/z/position`) 없이 `repo_id + slot_index + planet_type`만으로도 동작은 충분하며, 최소 필요정보 원칙으로 단순화한다.
- 프론트는 드래그 시 로컬 상태만 바꾸고 저장 버튼 클릭 시에만 DB 반영하는 흐름이 FE-2(상태/UX) 안정성에 맞다.

## 11) 기획·의사결정·학습 관점
- 팀에서는 스크럼/백로그 후속 정리 시 작업 동기화가 중요하며, WIP→DONE은 합의 기반으로 이동해야 한다.
- 바이브 코딩은 빠른 시행착오에 유효하지만 생성형 AI에 종속되면 안 되고, `질문/검증/이유 설명`이 핵심이다.
- 추천 시스템에서 새로운 기술보다 해결하려는 문제 정의가 우선이다. 기술은 수단이며, 면접/포트폴리오도 “왜 이런 기술을 선택했는지”가 핵심 평가 포인트다.

## 12) AI/데이터 분석 관련 배운 점
- 데이터가 거의 없을 때 콜드 스타트 대안으로 순수 가짜 데이터는 위험, 공개/기존 데이터 기반의 통계적 합성이 더 현실적이다.
- 추천은 데이터 성격 따라 전통 기법이 강할 수 있고, 딥러닝은 비정형(텍스트/음성/이미지)이나 대규모·멀티모달에서 유리하다.
- R은 통계 분석 친화형, Python은 운영/파이프라인 통합에 유리해 프로젝트 성격에 맞는 선택이 중요하다.

## 13) 확장성/가용성 계산 감각
- 가용성은 중단 없이 운영되는 시간 비율(예: 99.x%)이며, SLA는 사용자와의 운영 약속이다.
- 대용량 트래픽 추정 예시: `DAU 3억, 50% 활성, 사용자당 2개 트윗`이면 약 `3500QPS`.
- 미디어 포함 10%, 1MB 가정 시 일일 약 30TB, 5년 보관 시 약 55PB 규모로 추정된다.
- 설계면접에서는 정답 숫자보다 가정 수립, 계산 근거, trade-off(비용/성능/신뢰성) 설명 과정이 더 중요하다.
