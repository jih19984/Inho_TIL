# 2026-03-04 TIL

## 1) 빅데이터/아키텍처 확장 포인트 정리

### 1-1. 상태 없는 서비스(Stateless)와 확장
- 웹 서버는 무상태(stateless)로 운영하고, 상태는 별도 저장소에 둬야 다중 서버 운영이 안정적이다.
- 서버 1대 + DB 1대에서는 문제가 없을 수 있으나, 실제 서비스는 로드밸런서로 다중 웹서버 운영이 기본이다.
- 웹서버 간 상태 공유를 위해 **단일 상태 저장소/캐시/큐/DB**를 사용한다.

### 1-2. 캐시 정책
- 캐시는 요청량 급증과 DB 부하를 줄이는 핵심 수단이다.
- 교체 정책의 대표: `LRU`, `LFU`, `FIFO`
- 데이터 접근 패턴에 맞춰 정책을 고르면 캐시 효율이 올라간다.

### 1-3. 메시지 큐(MQ)
- 서비스 확장 시 컴포넌트 간 결합도를 낮추려면 비동기 경계 분리가 필요하다.
- 그럴 때 MQ 도입을 고려: `RabbitMQ`, `Kafka`
- 처리 단위 예시: 로그 수집, 메트릭 파이프라인, 알림/자동화 작업

### 1-4. 로그/메트릭/자동화
- 대규모 운영에서는 로그/메트릭을 서버별로 흩어놓지 말고 중앙 수집소에서 통합 관리.
- CI/CD를 통해 배포 자동화, 테스트 자동화, 롤백 자동화를 맞춰야 운영 안정성이 올라간다.
- DB/캐시/큐/애플리케이션 상태까지 모니터링이 필수.

### 1-5. DB 확장성: 수직/수평
- 수직적 확장(Scale-up): 하드웨어 업그레이드. 한계 존재.
- 수평적 확장(Scale-out): 샤딩이 핵심.
- 샤딩은 같은 스키마를 유지하면서 데이터를 분할 저장. 데이터는 중복되지 않음(레플리카와 다름).
- 예: `user_id % 4`로 4개 샤드에 분배해 DB 부하 분산.

### 1-6. 샤딩 운영 리스크
- `Re-sharding`: 기존 샤드 한계 초과 시 필요. 샤드 키/맵핑을 바꿔 재배치.
- `Hot spot`/`좋은 분포 불균형`: 특정 키에 트래픽 집중이 생기면 성능 저하.
- `Consistent Hashing` 같은 기법으로 집중/리밸런싱 문제를 완화한다.
- 샤딩 구조는 Join이 비싸고 난해해져, 비정규화(denormalization) 전략을 병행해야 할 때가 많다.

### 1-7. 핵심 정리
- 서비스 계층은 무상태화 + 다중화 + 캐시 + MQ + 로그/모니터링/자동화 + 샤딩 전략의 조합으로 확장한다.
- 정답은 하나가 아니라 단계별 제약(비용·일관성·운영 복잡도)에 맞춘 선택이다.

## 2) Twitter/고가용성 설계 감각

### 2-1. 가용성(Avaliability)
- 가용성: 시스템이 중단 없이 운영되는 정도(일반적으로 99%~100% 대역으로 표기)
- 비즈니스 약속(SLA)은 가용성, 응답시간, 장애보상 조건 등을 사용자와 계약한 값이다.

### 2-2. 트래픽/저장량 산정 예시
- 가정: DAU 3억, 50% 사용, 1인당 일일 2개 트윗 생성
- QPS(일일 생성 요청) 약 `3,000~4,000` 수준(`300M / 86400 ≈ 3472 QPS`)
- 미디어 포함 트윗이 10%라고 하면, 일일 미디어 저장량은 약 30TB(1MB/건 기준)
- 5년 보관 시 약 `55PB` 수준으로 증가(보존 정책/중복/압축에 따라 달라짐)

### 2-3. 면접 포인트
- 스케일/가용성 면접에서 중요한 건 단순 수치 암기가 아니라, 문제 풀이 절차다.
- 가정 세우기 → 단위 맞추기(초/일/년) → QPS/저장량 계산 → bottleneck 추정 → trade-off 제시가 핵심.
- 숫자는 근사치를 정당화할 수 있어야 설득력이 생긴다.
